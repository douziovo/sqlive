    In OpenAiCompatibleProvider.java:

    (a) Create a `private static String sanitizeErrorMessage(String message)` helper method:

    ```java
    private static String sanitizeErrorMessage(String message) {
        if (message == null) return null;
        return message.replaceAll("(?i)Authorization:\\s*Bearer\\s+\\S+", "Authorization: Bearer [REDACTED]");
    }
    ```

    The regex matches the Authorization header pattern case-insensitively and replaces the Bearer token value with `[REDACTED]`.

    Also add a secondary sanitization in the catch block that strips the actual API key value from the message: `message.replace(config.getApiKey(), "[REDACTED]")`. This catches cases where the API key appears outside the Bearer format.

    (b) In the `complete()` catch block (lines 107-111), replace the current logging:

    Current:
    ```java
    log.error("{} API call failed: model={}, elapsed={}ms", providerName, config.getModel(), elapsed, e);
    throw new AiProviderException("AI service call failed: " + e.getMessage());
    ```

    New:
    ```java
    String msg = e.getMessage();
    String sanitized = sanitizeErrorMessage(msg);
    if (msg != null) {
        sanitized = sanitized.replace(config.getApiKey(), "[REDACTED]");
    }
    log.error("{} API call failed: model={}, elapsed={}ms, error={}", providerName, config.getModel(), elapsed, sanitized);
    throw new AiProviderException("AI service call failed: " + sanitized);
    ```

    Key difference from the original plan: the raw exception `e` is NOT passed to `log.error()`. SLF4J's `log.error(msg, e)` calls `e.toString()` which includes `e.getMessage()` in its output -- this could leak the Authorization header. Instead, the sanitized message string is logged as a parameter, and only the sanitized message reaches the log output. The raw exception `e` is never passed to any log call.

    Note: the existing test `completeThrowsAiProviderExceptionOnHttpError` checks `ex.getMessage().contains("Connection refused")`. Since "Connection refused" does not match the redaction pattern, this test will still pass without modification.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Java heap -> log file | Structured log output could contain sensitive API keys if not sanitized |
| Backend thread -> AI provider network | Network I/O to external AI endpoints could hang indefinitely without timeouts |
| Runtime -> AiService.providers map | Concurrent reads during provider swap could see partial state without volatile + atomic assignment |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-INFO | Information Disclosure | OpenAiCompatibleProvider error logs | mitigate | Apply regex redaction of Authorization: Bearer header and api key string before logging; do NOT pass raw exception e to log.error() (SEC-02, Task 3) |
| T-01-DOS | Denial of Service | OpenAiCompatibleProvider WebClient | mitigate | Configure HttpClient with connect=5s / read=60s / write=30s to prevent thread exhaustion on unresponsive AI providers (AI-01, Task 1) |
| T-01-RACE | Tampering | AiService.providers field | mitigate | volatile keyword ensures visibility; synchronized refreshProviders() enables safe atomic swap (AI-02, Task 2) |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this plan; all changes are to existing Java source files |
</threat_model>
