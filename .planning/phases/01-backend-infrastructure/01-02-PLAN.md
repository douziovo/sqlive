      In `doFilter()`, the first time it runs for the application lifecycle, schedule the periodic cleanup task. Use a `private volatile boolean cleanupStarted = false` flag:

      ```java
      if (!cleanupStarted) {
          synchronized (this) {
              if (!cleanupStarted) {
                  cleanupScheduler.scheduleAtFixedRate(this::cleanupExpiredEntries, 5, 5, TimeUnit.MINUTES);
                  cleanupStarted = true;
              }
          }
      }
      ```
      Place this early in `doFilter()`, after the `path`/`clientIp` extraction but before the counter access logic. This avoids needing a `@PostConstruct` on a `Filter` implementation.

    (c) **Add cleanupExpiredEntries() method:**
      ```java
      private void cleanupExpiredEntries() {
          long now = System.currentTimeMillis();
          cleanupMap(aiCounters, now);
          cleanupMap(sqlCounters, now);
      }

      private void cleanupMap(Map<String, long[]> counters, long now) {
          counters.entrySet().removeIf(entry -> {
              long[] window = entry.getValue();
              synchronized (window) {
                  return now - window[0] > WINDOW_MS;
              }
          });
      }
      ```
      This correctly removes entries whose window has fully expired. The synchronized block ensures no concurrent `doFilter` thread is mid-update on this specific window.

    (d) **Lazy cleanup on access:**
      In the existing `if (now - window[0] > WINDOW_MS)` branch inside `doFilter()`, add a remove call so the stale entry is removed from the map:

      ```java
      synchronized (window) {
          if (now - window[0] > WINDOW_MS) {
              counters.remove(key, window);
              window[0] = now;
              window[1] = 1;
          } else if (...) { ... }
      }
      ```

      This is lightweight (one `remove` call per expired window) and provides lazy cleanup: every time an expired window is encountered, the old stale entry is removed from the map.

| RateLimitFilter -> ConcurrentHashMap | Counter entries can accumulate unboundedly if never cleaned, causing memory leak |
| SqlExecutionService -> DatabasePoolManager | Pool-full scenario needs graceful error propagation instead of silent hang or 500 |
