## Phase Scope

This is a tech-debt / reliability phase with 4 independent frontend fixes. No new features, no new dependencies, no new visual screens. The UI contracts below define **interaction states** and **copywriting** for the four scenarios.

| Fix | Component(s) | Visual Change | Interaction Change |
|-----|-------------|---------------|-------------------|
| FRONT-01 | `useInlineEdit.ts` | None (TypeScript only) | None |
| FRONT-02 | `sqlStatements.ts`, `useInlineEdit.ts`, `TableSection.vue` | Inline tooltip next to truncated cell | `enforceTypeConstraints` returns truncated metadata; `handleBlur` propagates to `TableSection` |
| FRONT-03 | `TableSection.vue` | Ghost row stays visible with red-tinted background on failure | Ghost row not cleared on submit until success confirmed via row-count watch |
| FRONT-04 | `DataVisualizer.vue` | None (component stays mounted) | `v-else-if` -> `v-show` for ER diagram tab |

## Copywriting Contract

### Truncation Warning (FRONT-02)

Shown as an inline tooltip when a VARCHAR/CHAR value is silently truncated during cell edit.

| Element | Copy |
|---------|------|
| Tooltip message | `值已截断：{originalLength} 字符 → {maxLength} 字符` |
| Inline cell indicator | Red exclamation icon (lucide `alert-triangle`, `h-3 w-3 text-destructive`) overlaid at top-right of the edited cell |
| Tooltip trigger area | The cell textarea itself -- tooltip appears on the next interaction (blur returns focus indicator) |

Flow:

1. User edits cell and blurs away
2. `enforceTypeConstraints` returns `{ value, truncated: { originalLength, maxLength } }`
3. `handleBlur` propagates truncated info to `TableSection.vue`
4. `TableSection` renders a small inline warning indicator near the cell
5. Tooltip appears on hover of the indicator

Error states:

| State | Behavior |
|-------|----------|
| Value fits within VARCHAR limit | No warning shown; no change to existing flow |
| Value exceeds limit | Warning tooltip shown; truncated value is submitted (existing behavior for compatibility) |

### Ghost Row Insertion Failure (FRONT-03)

Shown when the user submits a ghost row and the server returns an error.

| Element | Copy / State |
|---------|-------------|
| Ghost row visual | `opacity-100` (always visible on failure), with `bg-destructive/5` tint and `border-dashed border-destructive/30` top border |
| Ghost row textarea content | **Preserved as-is** -- all user input stays in the textareas |
| Submit button | Still visible; user can edit values and click again to retry |
| Success state | `ghostRow` cleared; row appears in table data; table scrolls to show new row (existing behavior) |
| Instruction label (optional) | Small `text-xs text-destructive` hint: `插入失败，检查数据后重试` rendered between the ghost row row and the last real row |

Flow:

1. User fills ghost row, clicks submit checkmark
2. `onGhostSubmit` emits `insert-row` event but does NOT clear `ghostRow`
3. `TableSection` watches `db.tables` for the target table's `rows.length` increase
4. If `rows.length` increased within the same reconciliation cycle -> success -> clear `ghostRow`
5. If `executionError` becomes non-null (or timeout) -> failure -> ghost row stays with `bg-destructive/5` tint and error hint

Edge cases:

| Situation | Behavior |
|-----------|----------|
| User navigates away from tables tab | Ghost row state is scoped to `TableSection`, which is preserved because parent `DataVisualizer` uses `v-if` for the `tables` tab. Not affected by FRONT-04 changes |
| Multiple ghost rows in different tables | Each `TableSection` has its own `ghostRow` reactive -- independent |
| User clears ghost row manually | User can delete textarea content; `hasGhostInput()` returns false; submit button hides |

### ER Diagram Position Fix (FRONT-04)

No copywriting changes. The `ErDiagram` component stays mounted when the user switches away from the ER tab and back. dagre positions computed in `onPaneReady` persist because the component lifecycle is not restarted.

| Element | Behavior |
|---------|----------|
| `activeTab === 'er'` | `v-show="true"` -> rendered (same as before) |
| `activeTab !== 'er'` | `v-show="false"` -> hidden but mounted (previously: destroyed) |
| Component instance | Single instance, lifecycle hooks run once |
| dagre layout | `onPaneReady` fires once on first render; positions persist in vue-flow internal state |

### Emit Type Safety (FRONT-01)

No UI copywriting. Pure TypeScript change to `useInlineEdit.ts`:

- Signature changes from `emit: (e: string, ...args: any[]) => void` to `emit: EmitFn` with generic constraint
- `TableSection.vue` consumers get type-checked `update-cell` event payloads
- No runtime behavior change

## Interaction States (Phase 3 Specific)

### State Machine: Truncation Warning (FRONT-02)

```
User types in cell -> blur happens
  -> enforceTypeConstraints(val, rawType)
    -> if fits: return { value }
    -> if truncated: return { value, truncated: { originalLength, maxLength } }
  -> handleBlur detects truncated field
    -> emit('update-cell', { tableName, oldRow, newRow }) (existing)
    -> propagate truncated info to TableSection via callback/ref
      -> TableSection renders inline warning indicator
```

### State Machine: Ghost Row (FRONT-03)

```
User fills ghost row -> clicks submit
  -> hasGhostInput() === true
  -> emit('insert-row', { tableName, newRow })
  -> ghostRow NOT cleared (changed from current behavior)
  -> watch target table rows.length:
    - if increases: ghostRow cleared (Object.keys cleared)
    - if executionError appears: ghostRow stays, bg-destructive/5 tint applied
```

### State Machine: ER Diagram (FRONT-04)

```
User clicks ER tab
  -> activeTab = 'er'
  -> v-show="true" => ErDiagram rendered
  -> onPaneReady fires => dagre layout computed, positions stored in vue-flow

User clicks another tab
  -> activeTab = 'indexes' (or other)
  -> v-show="false" => ErDiagram hidden but NOT destroyed

User clicks ER tab again
  -> activeTab = 'er'
  -> v-show="true" => ErDiagram visible with positions intact (no re-layout needed)
```
